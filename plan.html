Overview

The harness (~7K lines Rust+Go, 2-person team) and claude-code (~108K lines TypeScript, Anthropic production codebase) share the same core architecture: agentic LLM tool loop with human-in-the-loop approval, stdio/TUI two-process model, JSONL sessions, and OpenAI-compatible function calling. claude-code has 4 years of production hardening across every subsystem, offering a rich blueprint for improvements.

★ Self-Bootstrapping Plugin System centerpiece

Core principle: The harness must carry enough internal documentation that the AI agent can read, understand, and create plugins entirely by itself. When the user says "create a plugin that validates JSON before every file write", the agent reads the built-in plugin docs, scaffolds the directory, writes the hook script, and registers it — all without human intervention.

Design Goals

GoalHow
Agent-creatable All plugin docs injected into the system prompt. The agent uses its own read_file, write_file, bash tools to scaffold, test, and register plugins.
Dead simple One directory, two files minimum. Hook logic is a shell script or any executable. No SDK, no build step, no package manager.
Manageable Wire-protocol commands for install, enable, disable, remove, list. The TUI gets a plugin picker. The agent can call these commands itself via the stdio protocol.
Safe Plugins run as subprocesses with timeouts. Non-zero exit = deny. Paths are confined to workspace. Disabled plugins are never loaded.

  "version": "1.0.0",                 // semver

  "hooks": {
    "pre_write": {
      "script": "./hooks/pre_write",  // relative to plugin root, or absolute
      "timeout_ms": 5000,             // per-invocation timeout (default 5s)
      "pass_args": true               // pass tool args on stdin (default true)
    },
    "pre_bash": {
      "script": "./hooks/pre_bash",
      "timeout_ms": 30000
    }
  }
}

Hook Protocol — stdio JSON Contract

Every hook is a subprocess. The core writes a JSON object to stdin, waits up to timeout_ms, reads a JSON object from stdout. Non-zero exit or timeout = tool is denied.

stdin (what the core sends):

{
  "hook": "pre_write",           // which hook point fired
  "tool": "write_file",          // tool being invoked
  "workspace": "/home/user/proj",// confined workspace root
  "args": {                      // the tool arguments
    "path": "data/config.json",
    "content": "{ \"key\": 42 }"
  },
  "session_id": "abc123.jsonl",  // current session file
  "timestamp": 1719619200        // unix epoch seconds
}

stdout (what the hook prints):

{
  "allow": true,                 // false = deny the tool call
  "reason": "JSON is valid",     // shown to user on deny
  "modify": {                    // optional: mutate tool args
    "content": "{ \"key\": 42, \"validated\": true }"
  }
}

Hook Points — Full Catalog

HookFiresCan deny?Can modify?Typical use

pre_write Before write_file and edit Yes Yes — can transform content Validate JSON/YAML; format code; strip secrets
pre_read Before read_file, grep, glob Yes No Block access to sensitive paths
post_bash After any bash call No Yes — can annotate result Post-process output; trigger notification
post_write After write_file and edit No Yes — can annotate result Auto-format written files; run linter
post_read After read_file, grep, glob No Yes — can annotate result Inject license headers; redact secrets from output
session_start Once per session init No (fire+forget) No Set up environment; notify team channel

How the Agent Creates a Plugin (Self-Bootstrapping Walkthrough)

User says: "Create a plugin that validates every JSON file before writing"

Step 1 — Agent reads the plugin docs: The harness's system prompt contains the full plugin spec. The agent already knows the contract before the user even asks.

# Agent's internal knowledge (from system prompt):
# A plugin is a directory under .umans-harness/plugins/<name>
# with plugin.json + optional hooks/* scripts.
# Use the `write_file` tool, then `bash install-plugin` (send
# command to core via the TUI's /plugin-install slash command).

Step 2 — Agent scaffolds the plugin:

mkdir -p .umans-harness/plugins/json-validator/hooks

write_file .umans-harness/plugins/json-validator/plugin.json {
  "name": "json-validator",
  "version": "1.0.0",
  "description": "Validates all JSON files before writing. Rejects invalid JSON.",
  "hooks": {
    "pre_write": {
      "script": "./hooks/pre_write",
      "timeout_ms": 5000
    }
  }
}

write_file .umans-harness/plugins/json-validator/hooks/pre_write
#!/usr/bin/env python3
import json, sys
req = json.load(sys.stdin)
path = req["args"].get("path","")
content = req["args"].get("content","")
if path.endswith(".json") and content:
    try:
        parsed = json.loads(content)
        print(json.dumps({"allow": True, "reason": "JSON valid"}))
    except json.JSONDecodeError as e:
        print(json.dumps({"allow": False, "reason": f"Invalid JSON: {e}"}))
        sys.exit(1)
else:
    print(json.dumps({"allow": True}))

chmod +x .umans-harness/plugins/json-validator/hooks/pre_write

Step 3 — Agent installs it:

# Agent sends to core via the TUI slash command:
/plugin-install .umans-harness/plugins/json-validator
# → Core reads plugin.json, validates hooks, loads into state
# → Emits "plugin_installed" event to TUI
# → Hook now fires on every write_file/edit call

Step 4 — Agent verifies: The agent writes a test JSON file to confirm the hook fires.

Plugin State in Core (plugins field in State)

struct Plugin {

    description: String,
    enabled: bool,

}

struct HookConfig {
    script: PathBuf,                 // absolute path to executable
    timeout_ms: u64,                 // per-invocation timeout
    pass_args: bool,                 // send tool args on stdin
}

// In State:
plugins: RwLock<HashMap<String, Plugin>>  // name → plugin

Safety Guards

GuardRule
TimeoutEvery hook invocation has a hard timeout (default 5s for pre_*, 30s for post_*). Exceeded = deny (pre) or skip (post).
Exit codeNon-zero exit from a pre_* hook = tool call denied. Non-zero exit from a post_* hook = result unchanged, hook failure logged.
JSON parse failureIf hook stdout isn't valid JSON: treat as {"allow": false, "reason": "hook output parse error"} for pre_*, skip for post_.
Disabled pluginsHooks from disabled plugins are never invoked. State is not even loaded into memory.
Plugin validationOn install, core validates: plugin.json exists and parses, hook scripts exist and are executable. Failures → plugin_error event.
Workspace confinementPlugin scripts can reference files only within the workspace. Absolute paths are rejected (same workspace::resolve() guard as tools).
Hook failure isolationA broken hook never crashes the core. All hook subprocess failures are caught and logged.
